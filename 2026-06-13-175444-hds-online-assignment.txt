     Plan: LLM-powered diff summarization command

     Context

     The diff command produces structured ConfigDiff[] — field-by-field changes
     between two configs. The next step is a summarize command that feeds that
     diff (plus optional game context) into an LLM and returns a human-readable
     narrative. This turns raw field deltas into actionable release notes for
     LiveOps, QA, and engineering without requiring them to parse JSON diffs.

     The user wants:
     - A thin LLM abstraction layer in src/llm/ with a simple Message(prompt: 
     string): Promise<string> interface
     - Two concrete implementations: OpenRouter and LM Studio
     - The summarize command built on top of this abstraction
     - Optional --context flag accepting a game-description file to ground the
     LLM output
     - README updated with design decisions

     ---
     Architecture

     LLM abstraction layer: src/llm/

     src/llm/
       llm.interface.ts        — LlmClient interface with single send() method
       openrouter.client.ts    — OpenRouter implementation via @openrouter/sdk
       lmstudio.client.ts      — LM Studio implementation via REST (fetch)

     Interface:
     export interface LlmClient {
       send(prompt: string): Promise<string>;
     }

     Single method, single concern. The command layer doesn't know which backend
     is in use.

     OpenRouter client (openrouter.client.ts):
     - Uses @openrouter/sdk (already installed)
     - Constructor takes { apiKey, model } — fail with clear error if
     OPENROUTER_API_KEY unset
     - Calls openai.chat.completions.create({ model, messages: [{ role: "user", 
     content: prompt }] })
     - Returns choices[0].message.content

     LM Studio client (lmstudio.client.ts):
     - Uses fetch — no extra dependency
     - Constructor takes { baseUrl, model } (default baseUrl:
     http://localhost:1234)
     - POST /api/v1/chat with body { model, input: prompt }
     - Response: output[0].content (first message item in the output array)
     - No auth token needed for local LM Studio (or pass LM_STUDIO_API_TOKEN if
     set)

     Summarize command

     node dist/index.js summarize \
       --old <file> --new <file> \
       [--context <game-info-file>] \
       [--backend openrouter|lmstudio] \
       [--format text|json]

     New files:
     - src/commands/summarize.ts — commander command
     - src/services/summarize.service.ts — prompt construction + LLM call
     - src/types/summary.type.ts — SummaryReport interface

     New sample file:
     - inputs/game-info.md — short description of the game mechanics and key
     config fields

     Data flow

     1. readConfig() × 2 → diffConfigs() → ConfigDiff[] (existing code, no
     changes)
     2. Optionally fs.readFile(contextFile) for game description text
     3. Build prompt in summarize.service.ts
     4. Instantiate the right LlmClient based on --backend (default: openrouter)
     5. client.send(prompt) → raw string response
     6. Parse response into SummaryReport and print

     Prompt structure

     The prompt is built as a single string:
     You are a slot game configuration auditor...

     [Game context if provided]

     The following fields changed between two config versions:
     <ConfigDiff[] as JSON>

     Respond with JSON only: { "summary": "...", "highlights": ["...", "..."] }

     The service asks for JSON-only output and parses it. If parsing fails,
     wraps the raw string in { summary: rawText, highlights: [] } gracefully.

     ---
     Key decisions

     Why a separate src/llm/ folder:
     Future commands (risk-flagging, advisory notes) may also want LLM access.
     Centralizing the abstraction prevents each command from re-implementing
     HTTP calls.

     Why --backend flag instead of env-only selection:
     At the command layer, the user can explicitly choose local (LM Studio) vs
     cloud (OpenRouter) without touching env vars. Env vars supply credentials;
     the flag selects behavior.

     Why no --format summary on diff:
     Summarization is async, makes an external API call, and has its own
     --context option. Adding it as a format variant on diff would mix
     side-effectful behavior into a pure formatting switch.

     --context file vs hardcoded domain knowledge:
     Different teams configure different games. A file-based context is portable
     and reusable — the same game-info.md can be committed alongside the
     configs that describe that specific game.

      54 +## LLM summarization design decisions                          
      55 +                                                               
      56 +The `summarize` command feeds the structured `ConfigDiff[]` out

      58 +**Separate `summarize` command, not a `--format` on `diff`.**  
      59 +Summarization is asynchronous, makes an external API call, and 
         +has its own option (`--context`). Mixing it into the `diff` com
         +mand as a format variant would conflate pure formatting (synchr
         +onous, deterministic) with side-effectful I/O.                 
      60 +                                                               
      61 +**`--backend` flag (`openrouter` | `lmstudio`) rather than env-
         +only selection.**                                              
      62 +Env vars supply credentials; the flag selects runtime behavior.
         + This lets a user explicitly choose local (LM Studio, offline, 
         +no cost) vs cloud (OpenRouter) without modifying the environmen
         +t.

         + domain knowledge.**

         + but relies on field names alone.

     Context

     The diff and summarize commands each produce output independently. The
     report command combines both into a single markdown file suited for sharing
     with LiveOps, QA, or engineering — a ready-to-attach artifact that shows
     what changed and what it means, without the reader having to run multiple
     CLI commands.

     ---
     Command interface

     node dist/index.js report \
       --old <file> --new <file> \
       [--context <game-info-file>] \
       [--backend openrouter|lmstudio] \
       [--model <model>] \
       [--output <path>]

     - --old / --new — required, same as diff/summarize
